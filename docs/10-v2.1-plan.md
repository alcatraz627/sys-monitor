# v2.1 Plan

Scopes every open backlog item the user greenlit on 2026-06-18 ("Let's scope ALL
of these, I like them all"). Items are grouped by the *kind* of change they make,
not by priority — the build order at the end is dependency-aware and cuts across
the phases.

Each item carries: **what** it does for the user, the **anchor** (file:line the
work touches — grounded against the tree 2026-06-18, not assumed), an **approach**,
**effort**, **deps**, and a **verify** line (how it gets exercised, per the
project's exercise-before-done discipline — `--self-test` for math, `tools/drills/`
for runtime behavior, `docs/09-manual-checks.md` for pixels).

> Carried over from v2: the three items already SHIPPED — per-process disk I/O,
> per-process network I/O, battery panel element — are out of scope here. The
> battery *bar cell* (a menu-bar glyph, distinct from the shipped panel row) is
> item 7.4 below.

## Phase 6 — Useful while hidden

The monitor is currently only useful while the panel is open. These two make it
do work when you're *not* looking — the highest-value gap in the product.

| id | item | anchor | effort |
|----|------|--------|--------|
| 6.1 | **Threshold alerts.** Notify when CPU > X% or memory pressure ≥ warn (or a per-process runaway) persists for N consecutive ticks. The only feature that makes the monitor useful while hidden. New `AlertEvaluator` called from `publishSnapshot`; `UNUserNotificationCenter` for delivery. Per-metric *firing latch* + N-tick debounce + cooldown so it can't spam. Default OFF (settings-gated → couples to 9.5). | `SamplingCoordinator.swift:675` (hook), `AppDelegate.swift` (authorization at launch), new `Sampling/AlertEvaluator.swift` | ~90 lines |
| 6.2 | **Glyph tooltip / top consumer.** Surface the current top CPU consumer ("top: Chrome 84%") on the menu-bar item. Extend the existing `accessibilityValue` builder; also seed the right-click menu header. **Known platform limit:** macOS suppresses `toolTip` for nonactivating accessory apps (documented in v2 — that's why the panel has a footer status line) — so this lands as the accessibility string + menu header, NOT a hover tooltip. State that limit inline. | `GlyphRenderer.swift:183` (accessibilityValue), `StatusItemController.swift:112` | ~20 lines |

**Coupling:** 6.1's user-tunable thresholds are exposed by 9.5; build 6.1 with
sane hardcoded defaults (CPU > 85% for 5 ticks, pressure ≥ warn) so it's
shippable before the settings land, then wire 9.5 to override them.

## Build order (dependency-aware)

Cuts across phases — sequenced so each step is shippable and nothing builds on
unbuilt foundations:

```
1. 9.4 bar-cell reorder migration   ─┐ (unblocks 7.4; pure refactor, low risk)
2. 9.1 unify formatBps + bits/s      │  foundation / refactors first
3. 9.2 thresholds struct ───────────┘  (touches glyph + tokens; do before 6.1 reads it)

4. 6.1 threshold alerts  ──┐  highest user value; uses 9.2's plumbing
5. 9.5 alert thresholds  ──┘  (immediately after 6.1, same surface)

6. 7.1 disk-space row     ┐
7. 7.2 load-avg footer    │  cheap independent signals, parallelizable
8. 8.3 copy snapshot      ┘  (8.3 wants 7.1/7.2 data in the snapshot)

9.  7.4 battery bar cell      (needs step 1)
10. 8.1 watch-a-process
11. 6.2 glyph top-consumer
12. 8.2 global hotkey
13. 9.3 history-window  +  9.6 reset  +  Phase-9 second tier
14. 7.3 per-interface split   (needs real-network verification)
15. 10.1 / 10.2 spikes        (lowest certainty, last)
```

Rationale: the three refactors (9.4, 9.1, 9.2) are prerequisites for several
features and are the riskiest to retrofit later, so they go first. Alerts (6.1)
is the top user-value item and depends on 9.2's threshold plumbing. The cheap
signal rows fill the middle. The spikes (10.x) and the real-network-dependent
split (7.3) go last.

## Verification protocol (inherited from v2)

No item is "done" off inspection. Each closes via the cheapest real exercise that
induces its driving state:

- **Math / formatters** (9.1, 7.1 humanize) → `sys-monitor --self-test` boundary tables.
- **Runtime behavior** (6.1 debounce, 8.1 pin-survives-sort, 9.3 live re-trim) →
  a `tools/drills/<name>.sh` that induces the state and reads the log.
- **Pixels / real network** (every new row, 7.3) → appended to
  `docs/09-manual-checks.md` for the human-glance pass.

Mark un-exercisable rows `PENDING`, never `PASSED`. Per-process NET high-rate
attribution stays `UNCONFIRMED` until checked on an un-throttled network.
